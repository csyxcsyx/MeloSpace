# Java Web 端音乐网项目宪章

> 项目名称建议：\*\*TuneOrange / 橙音 Orange Music / LingTune 音乐网\*\*  
> 参考来源：用户提供截图中的《Java Web项目开发全程实录》“甜橙音乐网”板块。  
> 项目类型：Java Web 课程项目 / 可部署 Web 应用 / 音乐内容管理与播放平台。

## 0. 项目目的与落地口径（优化版）

### 0.1 一句话目标

本项目的核心目的，是在教材“甜橙音乐网”案例基础上，做出一个能体现 Java Web、数据库、文件上传、权限控制、播放器交互和服务器部署能力的现代音乐网站；它不追求商业音乐平台规模，而追求课程验收中“功能闭环完整、演示效果明显、部署可复现、文档说得清楚”。

### 0.2 项目真正要解决的问题

1. **课程能力展示**：用真实业务串联 Java Web 后端、数据库建模、前后端交互、权限控制、上传下载、异常处理和部署。
2. **产品体验升级**：把传统 CRUD 音乐网升级为有全局播放器、歌单、搜索、评论、收藏、推荐入口和现代 UI 的可演示产品。
3. **工程成果可交付**：最终交付的不只是页面截图，而是源代码、数据库脚本、部署配置、测试记录、演示脚本组成的完整项目包。
4. **风险可控落地**：音乐版权、推荐算法、UI 美化、部署环境是主要风险，必须通过 MVP 优先、授权素材、规则推荐和 Docker 化部署来控制。

### 0.3 推荐实施口径

优先采用 **Spring Boot REST API + Vue 3 + MySQL + 本地媒体存储/Nginx + Docker Compose**。若课程明确要求传统 Java Web 页面，可退回 **Spring Boot + Thymeleaf/JSP + MySQL**，但全局播放器与沉浸式交互会相对弱一些。

### 0.4 MVP 边界

MVP 必须先形成一个最小业务闭环：

|优先级|范围|验收口径|
|-|-|-|
|P0 必须|注册登录、角色权限、歌曲列表/详情/搜索、基础播放器、歌单创建与添加歌曲、管理员上传与上下架、Docker 本地部署|普通用户和管理员两条主流程可以完整演示|
|P1 应该|收藏、评论、最近播放、歌词显示、后台用户/评论管理、统一错误提示|提升项目完整度和稳定性|
|P2 可以|歌词高亮、场景推荐、拖拽歌单、沉浸播放页、封面主色背景|作为答辩亮点，必须在 P0/P1 稳定后再做|
|暂不做|商业平台音乐抓取、复杂机器学习推荐、多机高可用、支付会员、原生 App|避免偏离课程交付和合规边界|

### 0.5 成功判断

项目成功不以功能堆砌数量为唯一标准，而以以下结果为准：

* 用户能在 3 次点击内开始播放歌曲。
* 管理员能上传一首歌，并在前台被搜索、播放、收藏、评论。
* 歌单能创建、添加歌曲并保存。
* 系统能通过 Docker Compose 在新环境启动，数据库和媒体文件重启后不丢失。
* 至少 2 个创新交互可稳定演示。
* 项目文档能说明数据库设计、接口、部署、测试与素材来源。

## 2\. 项目背景

教材中的“甜橙音乐网”可以作为 Java Web 项目的基础案例，但其功能、交互与界面风格通常偏传统。当前音乐类 Web 产品更强调沉浸式视觉、全局播放体验、个性化推荐、歌单运营、搜索体验与跨设备响应式设计。

本项目将在保留 Java Web 课程核心能力的基础上，引入更完整的工程实践与更现代的音乐产品体验：

* 参考教材完成音乐网站基础业务闭环。
* 参考 Apple Music 的视觉氛围，但不直接复制其界面和素材。
* 加入全局播放器、沉浸播放页、歌词高亮、场景推荐、拖拽歌单等创新交互。
* 完成可运行、可部署、可维护的工程化成果。

## 3\. 项目目标

### 3.1 总体目标

构建一个基于 Java Web 的音乐网站，使用户能够浏览、搜索、播放、收藏歌曲，创建和管理歌单，并通过现代化 UI 与新颖交互获得更接近真实音乐平台的使用体验；管理员能够上传、编辑和管理音乐内容；系统能够稳定部署到服务器。

## 4\. 项目范围

* 在线支付、会员订阅、商业版权运营。
* 移动端原生 App。

### 4.3 可选扩展

* MusicKit 官方授权接入。
* MinIO 或云对象存储。
* Redis 缓存热门榜单。
* WebSocket 协同歌单。
* AI 生成歌单描述或情绪标签。

## 7\. 高层需求

### 7.1 功能需求

1. 系统应允许用户注册、登录和退出。
2. 系统应区分普通用户与管理员。
3. 系统应支持歌曲、歌手、专辑、歌单的增删改查。
4. 系统应支持音频、封面、歌词文件上传。
5. 系统应支持歌曲播放、暂停、进度拖动、音量调节、上一首、下一首。
6. 系统应支持播放队列和最近播放记录。
7. 系统应支持搜索歌曲、歌手、专辑、歌单。
8. 系统应支持用户收藏歌曲或歌单。
9. 系统应支持用户发布和删除自己的评论。
10. 系统应支持管理员管理用户、歌曲、评论和资源。
11. 系统应提供推荐、榜单或场景电台等扩展功能。
12. 系统应提供良好的错误提示和空状态页面。

### 7.2 非功能需求

|类型|要求|
|-|-|
|可用性|核心路径清晰，用户 3 次点击内可开始播放音乐|
|稳定性|歌曲加载失败不导致页面崩溃，接口异常有统一提示|
|安全性|密码哈希存储，后台接口鉴权，上传文件类型限制|
|性能|首页首屏资源适度压缩，列表接口分页，热门数据可缓存|
|可维护性|分层架构、统一异常、统一返回、接口文档清晰|
|可部署性|支持 Docker Compose 一键启动核心服务|
|合规性|不使用未授权音乐源，不抓取商业平台受保护内容|

## 8\. 高层技术方案

### 8.1 推荐技术架构

```text
前端：Vue 3 / Thymeleaf
后端：Java 17+ / Spring Boot / Spring MVC / Spring Security
数据库：MySQL 8.x
缓存：Redis，可选
存储：本地文件目录 / MinIO / 云对象存储
网关：Nginx
部署：Docker Compose
```

## 10\. 项目假设

1. 项目主要用于课程学习与展示，不进行商业化音乐运营。
2. 音乐素材来自自有、授权或可用于展示的测试素材。
3. 服务器资源为一台普通云服务器即可，配置建议 2 核 4G 或以上。
4. 并发规模以课程演示为主，不按大型商业平台设计。
5. 项目成员具备 Java、SQL、HTML/CSS/JavaScript 基础。
6. 若采用前后端分离，项目成员具备 Vue 或至少能使用组件化前端。

## 15\. 沟通与决策机制

* 每日或隔日同步一次进度：完成了什么、遇到什么问题、下一步做什么。
* 每个里程碑结束后进行一次小验收，确认是否进入下一阶段。
* 功能变更必须评估对时间和稳定性的影响。
* UI 调整不应影响核心功能上线。
* 所有关键配置、数据库脚本、部署命令必须写入文档。

\---

## 16\. 变更控制

当出现新增功能或技术变更时，按以下流程处理：

1. 描述变更内容。
2. 判断是否影响 MVP。
3. 评估开发时间、测试时间、部署影响。
4. 确定优先级：必须 / 应该 / 可以 / 暂不做。
5. 更新任务清单和文档。

优先级规则：

```text
必须：登录、歌曲、播放、歌单、后台、部署
应该：收藏、评论、搜索、最近播放
可以：歌词高亮、场景推荐、拖拽歌单、沉浸播放页
暂不做：复杂推荐算法、商业音乐平台抓取、多机高可用
```
